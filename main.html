1. 시스템 인프라 구성 (Infrastructure)

Client (Browser)

사용자 인터페이스 및
영지식 증명(Proof) 생성

snarkjs / circomlibjs

Supabase (DB)

사용자 식별 정보 및
Poseidon 해시값 저장

PostgreSQL Cloud

Render (Server)

Node.js 환경기반
영지식 증명 검증(Verify)

Node.js / Express

2. 데이터 흐름 및 보안 매커니즘

① 가입 단계
Password → [Poseidon Hash] → Supabase 저장 (비밀번호 유출 원천 차단)
② 인증 요청
ID 입력 → Supabase에서 저장된 Hash(Public Signal)를 브라우저로 로드
③ 증명 생성
Browser 내 연산: { Password, SavedHash, zKey } → zk-SNARK Proof 생성
④ 검증(Render)
Render Node 서버: snarkjs.verify(vKey, Proof) → 비밀번호 없이 본인 확인

3. 기존 방식 대비 보안 강점

보안 위협 기존 ID/PW 방식 본 프로젝트 (ZK-Auth)
서버 해킹 (DB 유출) 해시값 유출 시 무차별 대입 가능 해시가 유출되어도 Proof 없이는 로그인 불가
네트워크 가로채기 암호화된 비밀번호가 전송됨 비밀번호 정보가 전송 데이터에 0% 포함됨
사용자 프라이버시 서버가 모든 인증 정보를 소유 사용자가 증명의 주권을 가짐 (Self-Sovereign)
